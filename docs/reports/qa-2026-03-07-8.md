### Findings

#### Passed

- `backend/src/routes/alerts.py` exposes the new analyst-facing alert history and evaluation endpoints described by the MVP alert slice.
- `backend/src/storage/signal_store.py` persists alert history in a dedicated `alerts` table, keeping alert data separate from raw source observations and signal snapshots.
- `backend/src/services/alert_pipeline.py` evaluates current market opportunities and persists non-`HOLD` alerts, which is consistent with the current MVP focus on analyst-facing alerts rather than notification delivery.

#### Gaps / Deviations

- `docs/api/openapi.yml` defines `AlertHistoryResponse` with required fields `generated_at` and `alerts`, but `backend/src/models/schemas.py` defines `AlertHistoryResponse` with only `alerts`, and `backend/src/services/alert_pipeline.py` returns that narrower shape.
- `docs/api/openapi.yml` defines `AlertEvaluationResponse` with fields `evaluated_at`, `created_count`, and `alerts`, but `backend/src/models/schemas.py` defines `generated_at`, `evaluated_count`, and `alerts`, causing response-key drift.
- `docs/api/openapi.yml` defines `AlertRecord.id` as a string and allows status values `open`, `dismissed`, and `resolved`, but `backend/src/models/schemas.py` returns an integer `id` and restricts status to `open` only.
- The OpenAPI-traceable acceptance tests in `tests/acceptance/test_openapi_yml_acceptance.py` now fail on the alert contract drift above.

### Verdict

- [ ] PASS - implementation matches specs
- [x] FAIL - deviations listed above must be resolved before merge

### Recommended Actions

- Align backend alert response models with `docs/api/openapi.yml`, or update the OpenAPI schemas if the backend shapes are intentional.
- Re-run `python -m unittest discover -s tests/acceptance -p "test_*.py"` after the alert contract is synchronized.
- Keep the new alert-focused acceptance checks as a guardrail for future alert lifecycle changes.
